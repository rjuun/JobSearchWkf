## Purpose

Where [[B4. Translate Requirements to Areas of Expertise and Define JD Groups]] maps the Job Description to the candidate's own skills framework, this step interprets and prioritises the Job Description in its own language — extracting what the role actually demands, ranked by importance, and recording each requirement as a discrete, scoreable line item.

This step is primarily executed as a preparatory step for [[B6. Role Fit & Investment Worthiness Score]].
A "Standalone Output", as described in Sections D.1 and D.2, will only be produced when explicitly requested.

Output feeds the `Job Requirements` table in [[Job Hunting Lists.xlsx]]:
- `Lead`
- `Lead: ID`
- `Requirement Order` 
- `Rank` 
- `Requirement Group`
- `Requirement`
- `Requirement Description`
- `Skills`

## A. Source

Read the JD `.md` file from:
`OneDrive > Obsidian Vault > JobSearch Camunda > Job Descriptions`

The same file used in previous [[B4. Translate Requirements to Areas of Expertise and Define JD Groups]] step. No additional source required.

## B. Priority Groups

Each requirement is classified into one of three groups:

| Requirement Group | Definition |
|---|---|
| Core | Explicitly mandatory. The JD uses language such as "required", "must have", "essential", or positions it as a primary responsibility or qualification. Missing this requirement is a likely screen-out. |
| Important | Strongly preferred or frequently emphasised. The JD uses language such as "preferred", "strong advantage", "ideally", or repeats the theme across multiple sections. |
| Nice-to-Have | Optional, supplementary, or bonus. Mentioned once without emphasis, or framed as "a plus", "desirable", or "would be beneficial". |

**Ranking within groups:** within each group, requirements are numbered sequentially starting at 1 (`Rank`). The `Requirement Order` is a sequential global counter across all groups for the lead (Core first, then Important, then Nice-to-Have).

---

## C. Extraction Rules

1. Extract requirements from the entire JD — responsibilities, qualifications, requirements, and experience sections all contribute.
2. Each requirement should represent one distinct, addressable demand — not a section header or generic filler phrase.
3. Write the `Requirement` as a short descriptive label (5–10 words, title case).
4. Write the `Requirement Description` in the JD's own words or a faithful close paraphrase — never in the candidate's language.
5. Write `Skills` as the connected Skills/AoE codes from the framework (A–Q) with their full label and relevance indication, e.g.:
	- G. Process Management (core);
	- J. Change | Transformation Management (important)`
6. Aim for completeness over brevity — better to over-extract and consolidate than to miss a requirement that drives scoring.
7. **Language note:** 
	1. JDs written in German are extracted in German (`Requirement Description` may remain in German).
	2. The `Requirement` label should be written in English for consistency.

### D.2 End-of-Batch SharePoint Export

Produced once after all JDs in the batch have been processed.
Rendered as an interactive HTML widget with a Copy button.

**Format:** tab-delimited with RFC 4180-style quoting

**Column sequence** — must match `Job Requirements` in [[Job Hunting Lists.xlsx]]
 exactly:
- Lead
- Lead: ID
- Requirement Order
- Rank
- Requirement Group
- Requirement
- Requirement Description
- Skills

*(Leave blank: `ID` — auto-generated by SharePoint;
`Initial Match Strength`, `Initial Key Strengths`,
`Initial Missing / Weak`, `Initial Score` — future steps)*

**Rules:**
- One header row, one data row per requirement
- Rows grouped by Lead, sorted by Requirement Order ascending within each Lead
- Leads sorted by SharePoint ID descending (highest ID first)
- Multi-line content in `Requirement Description` or `Skills` enclosed in double quotes with internal newlines preserved
- Empty cells represented as empty strings (two consecutive tabs)
- Widget includes two buttons: "Copy all" (header + data) and "Copy data only"

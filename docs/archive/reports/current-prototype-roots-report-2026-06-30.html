Executive Verdict

The prototype has preserved the most important part of the manual process: a staged, gated journey from lead capture to screening, evidence review and CV output. The board, lead workspace, Career Graph and human Keep/Maybe/Drop gate are coherent and usable.

The main gap is fidelity, not architecture. Screening is relatively close to the documented process because B2-B6 load the process notes and B6 scoring is deterministic in code. Tailoring is much thinner: C2 uses token overlap, C3 mostly reuses existing bullet text, C5 is templated text, C6 generates a generic document, and C7 is a simple coverage metric. The first refactor pass materially improves that: C2/C3/C5/C7 now have live structured LLM paths and C6 has a real-template renderer. The remaining fidelity gap is now about scoping, validation, approval integrity and template completeness rather than missing step plumbing.

Recommendation: do not full-refactor the app yet. Keep the current architecture, choose one canonical workspace layout, then invest in methodology fidelity: wire C2-C7 prompts calibrate the new C-step live paths, inject B3 values context, restore traceability surfaces, and finish the real Word-template path without bypassing the human approval gate.

High lib/pipeline/tailoring.ts:53-120, 191-300, lib/ci.ts:15-26 C2/CV generation/CI guidance read profile evidence, skills, education, languages and tips globally. In a multi-tenant app, one user's evidence and feedback can influence another user's prompts or CV. Reviewer comment: before live multi-user use, pass ownerId through the pipeline and scope every read/write with it. Implemented: B/C pipeline reads and writes now receive an effective owner; pipeline_runs, llm_calls, CV variants, CI guidance and CV downloads are owner-scoped.
High app/api/ingest/route.ts and bookmarklet fetch behavior The ingest endpoint is public and fetch() from LinkedIn will not include app cookies by default. That means currentOwnerId() can fall back to the demo owner, so captured leads may land under the wrong account. Reviewer comment: use an authenticated capture token or include credentials with a non-wildcard CORS policy. Implemented: /roleproof/capture generates a signed capture token and /api/ingest rejects requests without it.
High lib/pipeline/tailoring.ts:89-96 Empty template slots are filled from the generic bullet bank/responsibilities. That bypasses the C2 human gate, so final CV content can include evidence the user did not approve for this role. Reviewer comment: blank or omit unapproved slots; do not fill them with fallback achievements unless the user explicitly approves a default set. Implemented: real-template slots now receive only Keep evidence; empty slots remain empty.
Medium lib/anthropic/schemas.ts:C2 and lib/pipeline/tailoring.ts:145-163 cvPosition is free text. If the model returns an invalid label, the approved bullet can silently miss every real-template slot. Reviewer comment: make cvPosition an enum of CV_SLOTS or normalize and validate before insert. Implemented: C2 schema uses the shared CV_SLOTS enum and the pipeline normalizes short codes before insert.
Medium app/actions/monitoring.ts:13-24 Mark-applied inserts a new application every call and updates a lead by id without owner/status checks. Reviewer comment: make this idempotent and scoped to the current owner; only allow transition from ready to applied. Implemented: mark-applied is owner-scoped and updates an existing application row when one already exists. Remaining: enforce allowed status transition if needed.

What Is Missing

Methodology fidelity

  • B3 does not inject the mandatory Values & Motives context.
  • C2-C7 process notes are not registered as runtime prompts. C2/C3/C5/C7 are now registered. C1/C4/C6 remain code-owned.
  • C2 evidence mapping is token overlap, not Sonnet reasoning over the full Career Graph. Live C2 now uses LLM mapping over broader owner-scoped evidence with validated CV slots.
  • C3 does not transform evidence into polished ATS bullets yet. Live C3 now rewrites Keep evidence. Needs live-output calibration against manual examples.
  • C4 lacks the intended 3-5 skill categories because skill_category is not modeled.
  • C5 is template text rather than a real tailored profile generation step. Live C5 exists, but the real-template DOCX currently does not insert that profile text.
  • C7 rating is not an Opus assessment. Live C7 exists and stores the report output in pipeline_runs; rich lead-level persistence is still missing.
  • Real-template C6 fills only the 11 Professional Experience placeholders; profile and skills are not filled into the Word output.
  • Unapproved fallback evidence can still enter empty real-template slots. C6 now leaves empty slots blank unless supported by Keep evidence.
  • B7 summary and cover-letter generation are absent.

Product and operations

  • No active CI dashboard view for the seeded initiatives, cost panel and process backlog.
  • No in-app indication of LLM_MODE=mock versus live. Header badge added.
  • No expert trace view showing step IDs, prompt source and model. Compact trace disclosures are now folded into the regular workspace. Evidence refs are visible in triage; raw output JSON remains hidden.
  • No deployment-aware bookmarklet generator. Origin-aware, signed-token bookmarklet added.
  • No application log. Mark-applied scaffold added. Follow-up tracker, outcome UI and target-company monitoring remain.
  • New live/deep pipeline reads and writes are not consistently scoped to currentOwnerId(). Main B/C, CI and D paths are now owner-scoped. Keep auditing new surfaces as they are added.
  • No robust e2e tests beyond verifier scripts.
  • No visual CV preview/PDF render check before download.

Recommended Paths Forward

Recommended

Faithfulness First

Best if your goal is to trust the app as a replacement for your manual process.

  • Fix owner scoping across C2-C7, CI guidance, CV variants and applications.
  • Wire B3 values context.
  • Register C2-C7 prompts and schemas. Calibrate C2/C3/C5/C7 live outputs against manual examples.
  • Run live LLM tests on 3-5 known leads.
  • Add expert trace surfaces. First pass folded into regular panels; add deeper rationale/output drawers next.
  • Move to the real CV template. Finish profile/skills fidelity and render preview.

Product First

Best if the goal is to show a polished SaaS journey quickly.

  • Consolidate RoleProof branding. Continue cleaning historical-only docs over time.
  • Keep guided onboarding front and center.
  • Deploy with Supabase Auth and RLS.
  • Harden bookmarklet auth attribution before multi-user deploy. Signed capture token added.
  • Add application tracking. Expand the scaffold into outcome/follow-up tracking.
  • Keep mock C-tailoring explicit. Header badge and per-run trace added.

Full Refactor

Only worth it if live prompt wiring still cannot reach manual quality.

  • Make pipeline_runs the UI source of truth.
  • Build a step registry mirroring Process/.
  • Use graph search/RAG for evidence retrieval.
  • Unify board/workspace/apply flow.
  • Consider Camunda runtime only after the process stabilizes.

Prioritized Next Steps

  1. Fix data isolation first: Main B/C/CI/D paths are owner-scoped; keep auditing any new surface before deploy.
  2. Fix bookmarklet identity: Signed capture token implemented on /roleproof/capture and /api/ingest.
  3. Protect the human gate: C6 no longer fills unapproved fallbacks and cvPosition is validated against the real slot list.
  4. Protect orientation: RoleProof routes, Keep/Maybe/Drop vocabulary, LIVE/MOCK badge, alternative-preview badge and workspace trace are in place.
  5. Close the screening gap: inject Values & Motives into B3 and verify B1-B6 against hand-calculated examples.
  6. Calibrate live tailoring: compare C2/C3/C5/C7 output against existing manual C-step rows and tune prompts/schemas.
  7. Finish CV fidelity: decide how profile/skills enter the real template, group bullets by role/position, and add a document render check.
  8. Recover operations: restore CI initiatives/tips/usage/application backlog into RoleProof-native surfaces.
  9. Deepen trust surfaces: add per-requirement rationale, output JSON and token/cost detail where helpful.
  10. Then deploy: Supabase Auth/RLS, storage validation and a concise demo script.

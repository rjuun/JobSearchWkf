# O4 · Multi-tenant (sign up → your own isolated graph)

> Detailed record of phase **O4** from the [user-journey roadmap](../design/USER_JOURNEY.md).
> Friendly review (with screenshots): [`o4-multitenant.html`](o4-multitenant.html).

O1–O3 built the Career Graph for one person. **O4 opens it to many** — anyone can sign up and gets
their own, empty, fully isolated graph. The plan's O4 is "Supabase Auth + RLS"; this environment has
no Supabase (local Postgres only), so O4 ships as **real multi-tenant on the existing stack** — the
exact `owner_id` model that maps 1:1 to Supabase RLS on deploy.

## What was built

| Piece | File |
| --- | --- |
| `users` table (+ migration 0003) | `lib/db/schema.ts`, `drizzle/0003_silent_sabretooth.sql` |
| DB-backed auth: scrypt hashing, session carries the user id, **`currentOwnerId()`** | `lib/auth.ts` |
| Sign up + login actions | `app/actions/auth.ts`; `/signup` page; middleware allowlist |
| **Per-user scoping** — every read and write | `lib/queries.ts` (reads) + the action writes (profile, onboarding, coaching, tips, leads, ingest) |
| Demo user (id = `DEMO_OWNER_ID`, owns the seeded data) | `scripts/seed-users.ts` |
| RLS policies for Supabase | `drizzle/rls.sql` |

## The scoping primitive

`currentOwnerId()` reads the session and returns the logged-in user's id. **Every query is scoped to
it**, and every user-data write sets `owner_id` to it. So:

- A new sign-up gets a fresh `profiles` row and **sees nothing of anyone else** — empty graph, empty
  board, empty dashboard.
- `getLead(id)` is owner-scoped too, so one user can't load another's lead by guessing its id.

This is precisely the Supabase model: `owner_id = auth.uid()`. On deploy, the local `users` table is
replaced by Supabase's managed `auth.users`, and [`drizzle/rls.sql`](../../drizzle/rls.sql) enables
RLS with an `owner_id = auth.uid()` policy on every owner-scoped table — turning the app-layer
guarantee into database-enforced, defence-in-depth isolation.

## Key decisions

- **Built + verified locally**, not stubbed. Real users, real signup, real isolation — provable here.
  The Supabase swap is auth-provider + RLS, not a re-architecture.
- **scrypt, no new dependency** (Node's `crypto`).
- **A backstop default.** `owner_id` still defaults to `DEMO_OWNER_ID` at the schema level, so any
  not-yet-scoped insert pools into the demo account rather than leaking across real users.

## Verification

- `tsc --noEmit` + `next build` green (`/signup` builds; 22 routes).
- **Isolation proven** (`scripts/verify-o4.mjs`): the demo user sees the full 100/Comprehensive
  graph; a brand-new sign-up ("Alex Rivera") lands on an **empty graph (Getting started)** — 0
  positions / stories / skills — while the demo's evidence (6 / 25 / 7) is untouched. Confirmed in
  the DB and on screen. Test user removed afterwards.

## Gaps · missing · future

- **Deploy on Supabase Auth.** Swap the local `users`/password flow for Supabase Auth (magic link /
  SSO) and apply `drizzle/rls.sql`. Email verification, password reset and rate-limiting come with it.
- **Pipeline-internal writes.** Screening/tailoring inserts (job_requirements, requirement_tailoring,
  llm_calls, pipeline_runs) still rely on the `DEMO_OWNER_ID` default — so a *new* user who runs the
  full pipeline would have those rows pooled to the demo (only minor cosmetic effects: requirement
  counts on the board, usage on the dashboard). Scope them to the lead's owner next.
- **No org / team tenancy** — it's one graph per user; shared workspaces are a later model.
- **The bookmarklet `/api/ingest`** captures to whoever's session cookie is present (else the demo);
  a per-user capture token would be cleaner.

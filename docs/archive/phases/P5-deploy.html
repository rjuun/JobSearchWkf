Phase 5 · Deploy

Hardening & share prep

Everything needed to put the prototype behind a shareable, auth-gated URL — a Supabase-ready storage backend, a health probe, server-only secrets, and a concrete deploy runbook.

Code-ready · provisioning is the owner's step

1Production hardening

Storage

Supabase or filesystem, by env

Storage switches to a Supabase bucket when SUPABASE_URL + SUPABASE_SERVICE_ROLE_KEY are set (Vercel's disk is ephemeral); local dev stays on the filesystem. Same API.

Probe

/api/health

Public readiness check → { ok, db, storage, llm } with 200/503, so you can see at a glance that prod is wired correctly.

Secrets

Server-only, secure cookies

Service-role and Anthropic keys never reach the browser; the session cookie is secure in production; all LLM calls run server-side.

Live LLM

One env switch

LLM_MODE=live + ANTHROPIC_API_KEY flips every step from deterministic mocks to real Claude calls — no code change.

2The deploy runbook

  1. Supabase project — copy the Postgres URL into DATABASE_URL; create a private jobsearch bucket; set the Supabase env vars.
  2. Schema — npm run db:migrate against the prod database.
  3. Seed — locally, with prod env, npm run seed (your real workbooks → prod Postgres + Storage).
  4. Secrets — set APP_EMAIL/PASSWORD, a strong SESSION_SECRET, LLM_MODE=live, ANTHROPIC_API_KEY in Vercel.
  5. Vercel — import the repo (Next.js auto-detected), add env vars, deploy.
  6. Share — add a domain; the login middleware already gates everything, so a shared link shows a sign-in wall. Check /api/health.

3Verified vs. needs your accounts

  • ✓ Verified here: build with the Supabase client bundled; the filesystem storage path (CV generates & downloads); the health route registered.
  • Needs your cloud: the live Supabase Storage round-trip, live Anthropic calls, and the Vercel deploy — all in place, selected by env.

4Recorded deviations

Drizzle migrations (not the Supabase CLI) — Supabase is just the Postgres host. Custom single-user auth (not Supabase Auth): the app connects with a privileged role, so RLS isn't the enforcement boundary yet — app-layer owner_id scoping is. Going multi-user means adopting Supabase Auth + RLS (the schema already carries owner_id). Cost: Anthropic usage is the only real spend — single-digit dollars for a demo; Supabase + Vercel free tiers suffice.

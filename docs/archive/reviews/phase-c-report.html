Issue 7 One live draft per prompt

Every time you re-drafted an answer, a new row was inserted and the old one was never removed. Edit a prompt five times and you'd leave five answer rows behind (approve just took the newest).

Fix: before inserting a fresh draft, delete the prompt's prior un-committed drafts. Committed answers — the ones carrying a committedNodeId, i.e. the record of a real approval — are left untouched.

await db.delete(coachingAnswers).where(and(
  eq(promptId), eq(ownerId),
  isNull(committedNodeId)   // only clear drafts,
));                          // never a committed answer

Issue 10 Name the magic number

The strength meter capped scores at a bare 99 buried in one line. That 99 is meaningful — it's why the meter can never hit 100 (the "never-done" model). A lone literal hides that intent.

Fix: extracted export const MAX_STRENGTH = 99 with a comment explaining it's the asymptotic cap, and used the constant at the cap site.

Pure readability — no behaviour change — but now the one number that encodes a product principle is named and documented.

Issue 11 Coach approvals now count as "decisions before win"

The dashboard shows "Decisions before win" — how many evidence decisions you made between your first paste and your first generated CV. But it only counted C2 tailoring Keeps. A user who did most of their work in the coach (approving answers) and then generated a CV could see "0 decisions" — plainly wrong, and it made the activation numbers untrustworthy.

Fix: a coach approval now emits a distinct 'coach_approval' activation event, and the metric counts both kinds as decisions. Crucially, the separate "Evidence kept" number stays C2-only, so the two stats keep their distinct meanings.

// a "decision" = a C2 Keep OR a coach approval
const DECISION_KINDS = new Set(['keep', 'coach_approval']);
decisionsBeforeWin = events.filter(e => DECISION_KINDS.has(e.kind) && between(start, cv)).length;
totalKeeps         = events.filter(e => e.kind === 'keep').length;  // unchanged — C2 only
Why a new event kind, not reuse 'keep'? If a coach approval recorded a 'keep', it would silently inflate the "Evidence kept" number, which specifically means C2 tailoring Keeps. A distinct kind lets one metric widen (decisions) while the other stays precise (keeps). Honest signals over convenient ones.

Issue 9 No more silent no-op on approve

If you double-clicked "Approve", or clicked it on a prompt a background refresh had already resolved, the server correctly did nothing — but it returned nothing either way, so the UI still ran its optimistic "approved!" animation over an action that never happened. Confusing, and it could briefly desync the rail.

Fix: approveAnswerAction now returns a structured result:

type ApproveResult =
  | { ok: true }
  | { ok: false; reason: 'nodraft' | 'already' | 'invalid' };

The coach rail reads it: on ok, it runs the normal capture + advance. On a no-op, it shows a small self-dismissing notice — "Already approved" or "Draft the evidence first" — and just refreshes, without the misleading optimistic update. Double-clicks and races are now visible, not mysterious.

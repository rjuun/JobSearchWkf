Issue 5 · The strength meter now rewards coaching

The problem

Coverage was frozen at screening time

When you screen a job ad, the LLM scores each requirement (0–10) and labels a match strength. The "Relevancy" slice of your strength meter — a full 25 of 100 points — was computed purely from those initial numbers. The coach would then ask you to close a gap, you'd answer, evidence would land in your graph… and the meter would not move, because it only ever looked at the frozen screening scores. Worse, the coach would keep re-asking about a requirement you'd already covered.

Why this mattered M1's whole premise is a "compounding, never-done" meter — the felt reward that makes you keep coaching. A meter that ignores coaching quietly breaks that promise.
The fixlib/career-graph.ts

Recompute coverage against the current graph

Three new pure (no-database, no-LLM, fully testable) helpers:

  • evidenceTokens(graph) — flattens all your evidence (stories, actions, results, responsibilities, skills, positions, bullets) into a set of individual words.
  • salientTerms(requirement) — extracts the meaningful words of a requirement, dropping stopwords ("the", "and", "across", "led"…) and very short filler.
  • graphCoversRequirement(req, tokens) — returns true when ≥ 60% of a requirement's salient terms appear in your evidence tokens.
export function graphCoversRequirement(requirement, tokens) {
  const terms = salientTerms(requirement);      // meaningful words only
  if (terms.length === 0) return false;      // all-stopword req → never "covered"
  const hits = terms.filter(t => tokens.has(t)).length;
  return hits / terms.length >= 0.6;         // strong majority present
}

Two deliberate design choices worth understanding:

1. It ORs with the screening verdict — never overrides it. A requirement counts as covered when screening judged it not-weak OR the graph now covers it. The screening result stays a floor; live coverage can only add. So flagging a fresh, uncovered target still correctly shows as "open."

2. It's biased to under-claim. The 60% threshold and whole-word matching mean it would rather miss a real match than invent a fake one — matching the project's "truthfulness over optimisation" rule. It matches whole words, so "board" in a requirement does not match "dashboard" in your evidence.

Wired into two placesqueries.ts · coaching-queue.ts

The meter and the coach both use it

The meter (getTargetCoverage): now loads the target requirements and counts each as covered if not-weak-at-screening OR graphCoversRequirement(...). Because it runs while the graph is being assembled, it sees exactly the evidence you have right now.

The coach (generatePrompts): filters out target requirements the graph already covers before generating prompts — so a gap you've coached stops being re-asked. The now-covered prompt "parks" (goes inactive) on the next regeneration, which keeps it reversible if you later un-flag the role.

The neat consequence Approve a coached answer → your evidence graph gains words → next time the meter/coach load, those words satisfy a requirement's salient terms → relevancy ticks up and the prompt retires. No extra bookkeeping, no new database columns. The graph is the source of truth.

Issue 6 · The proof trail now tells the truth

The problemlib/pipeline/tailoring.ts

Every CV line said "Imported", even coached ones

The M7 proof trail labels each tailored CV line by where its evidence came from: Imported / Coached / Swapped. But the column that drives that label (provSource) was never set on insert — so it always fell back to its database default, 'imported'. A line backed by evidence you built in the coach still rendered as "Imported." The trail looked complete but was cosmetic.

The fix

Carry the source, stamp the label

Every evidence node already records its own source (imported, authored, or ai_coached — the latter set whenever the coach commits an approval). The gap was simply that this fact was dropped on the way into the tailoring table. Two small changes reconnect it:

// 1. gatherEvidence now carries each node's source
out.push({ ref: a.refCode, kind: 'STAR action', ..., source: a.source });

// 2. the C2 insert stamps the label from that source
function provFromSource(source) {
  return source === 'ai_coached' ? 'coached' : 'imported';
}
rows.push({ ..., provSource: provFromSource(ev.source) });

Now a CV line whose evidence you grew in the coach renders as "Coached" in the proof trail — the M7 "clickable, computed provenance" surface finally reflects reality. (The third label, "Swapped," belongs to an evidence-swap flow that isn't built yet; that stays out of scope here.)

Phase B knowledge check

Six questions. You need 5 / 6 (83%) to pass. Answers explained on submit.

Q1. Before Phase B, why didn't coaching a gap ever move the "Relevancy" slice of the strength meter?
Frozen input. The evidence was saved — but coverage read only the initial screening scores, and nothing recomputed them, so new evidence had no path to the meter.
Q2. A requirement counts as "covered" when it was not-weak at screening OR the graph now covers it. Why OR rather than replacing the screening verdict entirely?
Additive, not destructive. The keyword heuristic is weaker than the LLM's judgment, so it's used only to add coverage. ORing preserves the screening result as a floor.
Q3. The heuristic requires ≥60% of a requirement's salient terms to appear, and matches whole words. What is this deliberately biased toward?
Honest bias. A strict threshold + whole-word matching means false "covered" is unlikely. Under-claiming coverage is the honest direction for this product.
Q4. Requirement "board reporting"; your evidence says "Built the executive dashboard suite." Covered? Why?
Whole words only. Evidence is tokenized into whole words, so "dashboard" is its own token and never satisfies the term "board". Neither salient term is present → not covered.
Q5. After you approve a coached answer that covers a target requirement, what happens to the coach prompt for that requirement on the next regeneration?
Park, don't delete. generatePrompts drops now-covered target reqs; the open prompt parks (inactive) rather than being destroyed, so un-flagging the role can bring it back.
Q6. For Issue 6, why did every tailored CV line render as "Imported" before the fix?
Unset → default. The evidence node already knew it was ai_coached, but that fact was dropped before the insert, so provSource took its database default of 'imported'. The fix carries the source through and stamps the right label.

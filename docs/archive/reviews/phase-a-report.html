Phase A — Atomic Undo & Resilient Recovery

Every fix in this phase answers one question: if this operation fails halfway through, what state does the user end up in? The review found four places where the answer was "a broken one." Below is what each was, the intuition for why it mattered, and exactly what changed.

The one idea to hold onto A user action like "undo my last answer" isn't one database write — it's five or six. Delete the follow-up, revert the star, flip two statuses, re-score. If the server dies after step 3, you've got half an undo. The fix is a transaction: a wrapper that says "do all of these, or none of them." That single concept is 3 of the 4 fixes.

Concept primer (30 seconds)

Transaction

A database transaction groups several writes so they commit together or roll back together. Nothing outside the transaction sees a half-finished state, and if any step throws, everything already done inside is undone automatically. In this code it's db.transaction(async (tx) => { … }) — and crucially, you must use the tx handle for every write inside, not the global db, or that write escapes the safety net.

Executor threading

To let a shared helper (like "revert this star") run either standalone or inside someone's transaction, you pass it the handle to use. We call that a DbExecutor. Both the global db and a transaction's tx satisfy the same shape, so the helper doesn't care which it got — it just uses what it's handed.

Best-effort (fire-and-forget)

Some steps are bookkeeping — recording a strength snapshot, regenerating the prompt queue. If the important write already committed, a failure in bookkeeping should log and move on, never blow up the user's request. The trick is that the bookkeeping must be idempotent (safe to re-run), which the queue regeneration already is.

The four fixes

Issue 3 · enabler lib/coaching-commit.ts

Make revertCommit able to join a transaction

This one fixes nothing by itself — it unlocks Issue 2.

The revertCommit function undoes a graph change (delete the coached star, restore the old summary, etc.). But it was hardwired to the global db — so even if its caller opened a transaction, the revert's deletes ran outside it and couldn't be rolled back. We gave it an optional executor argument, defaulting to db so existing callers are unchanged:

// before — always the global db, can't participate in a tx
export async function revertCommit(owner, plan) {
  await db.delete(starResults)...

// after — takes the handle to use
export async function revertCommit(owner, plan, database: DbExecutor = db) {
  await database.delete(starResults)...

We also added asRevertPlan() — a small validator that checks the revert instructions read back from the database actually have the shape we expect before acting on them. These are written by our own code, so this is a seatbelt against schema drift or a hand-edited row, not a security boundary. An unrecognized shape returns null → we skip the revert instead of crashing on undefined.length.

Issue 2 · highest priority app/actions/coaching.ts · undoApprovalAction

Make "undo" all-or-nothing

The failure it prevents: Undo used to run as a bare sequence of writes. Picture a user undoing an answer that had spawned a follow-up question:

purge follow-up's stars→ delete follow-up prompt→ 💥 crash / refresh→ revert main star→ flip status

Stop at the 💥 and you've deleted the follow-up but not reverted the original — an orphaned star, a prompt stuck in the wrong state, and a strength score that no longer matches reality. A concurrent approve landing at the same moment could interleave and corrupt it further.

The fix: the entire unwind now runs inside one db.transaction. Every write uses tx (including the newly-threaded purgePromptEvidence and revertCommit). Either all of it lands or none of it does.

const didUndo = await db.transaction(async (tx) => {
  // find the answer; if none, bail cleanly (empty tx, no-op)
  for (const c of children) await purgePromptEvidence(owner, c.id, tx);
  await tx.delete(coachingPrompts)...      // child follow-ups
  if (plan) await revertCommit(owner, plan, tx);  // this approval's star
  await tx.update(coachingAnswers)...        // null out the commit
  await tx.update(coachingPrompts)...        // back to 'drafted'
  return true;
});
if (!didUndo) return;
await snapshotStrengthAndRegen(owner);       // bookkeeping, after commit
Subtle but important The strength snapshot and queue regen happen after the transaction commits, not inside it — because they re-read the graph through the global db, which can't see the transaction's uncommitted writes. Doing them after commit means they read the correct, reverted state. This mirrors how approve already worked.
Issue 1 · medium app/actions/coaching.ts · approveAnswerAction

Stop the approval "tail" from erroring the request

Approving an answer already had a proper transaction around the important part (claim the prompt, write the star). But after that committed, three more steps ran unguarded: maybe spawn a follow-up, snapshot strength, regenerate the queue. If any threw — a transient DB blip, a bug in queue generation — the user saw an error even though their evidence was safely saved.

The fix wraps that tail in try/catch that logs instead of throwing, extracted into a shared snapshotStrengthAndRegen helper (now reused by both approve and undo). This is safe precisely because the queue regeneration is idempotent — if it fails now, the next coach-page load regenerates it correctly.

try { await maybeSpawnFollowup(...); }
catch (err) { console.error('[coaching] follow-up spawn failed', err); }
await snapshotStrengthAndRegen(owner);  // itself try/catch + log inside
Issue 4 · medium app/actions/onboarding.ts · startScreenFirstAction

Don't mark onboarding "done" before the lead exists

The failure it prevents: the cold-start flow is "paste CV + job ad → screen it → land on a verdict." The old order committed the imported career graph and flipped onboarding to done before it created the lead:

extract graph→ commit graph + onboarding=DONE→ createLead→ screen

If createLead threw, the user was left with a "done" onboarding and an imported graph but no lead to land on — and, since createLead wasn't guarded, a raw error page instead of a graceful retry. The fix reorders so the lead is captured before the graph commit (the "done" flip), wrapped so a capture failure redirects to /start?error=1 with onboarding still in reviewing:

extract graph→ stage + createLead (guarded)→ commit graph + onboarding=DONE→ screen (best-effort)

Capture doesn't need the graph (it just files the job ad), and screening still runs after the commit so it sees the imported history. The user can only reach a "done" state once there's genuinely a lead waiting for them.

Phase A in one breath

  • Issue 3 — revertCommit can now take a tx handle (unlocks 2).
  • Issue 2 — undo is now one atomic transaction: no orphaned stars.
  • Issue 1 — approve's after-save bookkeeping is best-effort, can't error the request.
  • Issue 4 — cold-start captures the lead before flipping onboarding "done".

Common thread: define the state you're in when a step fails, and make it a recoverable one.

Phase A knowledge check

Six questions. You need 5 / 6 (83%) to pass. Answers explained on submit.

Q1. What is the core purpose of wrapping the undo logic in a db.transaction?
Atomicity. A transaction's whole point is all-or-nothing. It's not primarily about speed, isolation from readers, or retries — it's that a crash mid-way leaves no partial change behind.
Q2. Why did revertCommit need an optional DbExecutor parameter added before the undo transaction could work?
You must use the tx handle for every write. If revertCommit kept using the global db, its deletes would escape the transaction — commit even if the surrounding tx rolled back. Threading the executor lets it use tx.
Q3. Why do the strength snapshot and queue regeneration run after the transaction commits rather than inside it?
Visibility. Reads through the global db can't see a transaction's uncommitted writes. Snapshotting inside the tx would record the old strength. Running after commit reads the true post-undo state.
Q4. In Issue 1, approve's after-save steps were made "best-effort" (try/catch + log). Why is that safe to do here?
Committed + idempotent. Swallowing an error is only safe when nothing important is lost. The evidence was already saved in the tx, and regenerating the queue can be safely re-run — so logging and moving on is correct.
Q5. What bad end-state did the Issue 4 reordering specifically prevent?
Orphaned "done" state. Committing the graph and flipping onboarding to done before creating the lead meant a capture failure stranded the user in a finished-looking state with nowhere to go. Capturing the lead first fixes it.
Q6. What does asRevertPlan() do, and what is it not?
Shape check. Revert plans are written by our own commit code, so validating them guards against schema drift or a corrupt row (avoiding a crash on undefined.length) — it isn't defending against a malicious user, since users never author these.

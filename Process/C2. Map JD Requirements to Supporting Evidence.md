## Purpose
Map every requirement extracted in Step [[B5. Extract Requirements from Job Description]] to a specific, verifiable evidence from the candidate’s background.
This step creates the traceability foundation required for writing truthful, ATS-optimized CV bullets, Skills and Profile in subsequent steps (C3, C4, and C5).

The output is a SharePoint-ready table that links each Job Description requirement to the exact source of evidence (STARs, Responsibilities, Education, or Languages) and indicates where the corresponding bullet should appear in the CV.

---

## A. Execution Sequence

1. Retrieve the list of all requirements for the Job Lead from the Job Requirements table (output of B5).
2. For each requirement, search the Profile_Reference_Workbook.xlsx for satisfactory evidence that the candidate fulfils it.
3. Construct the **Reference** column using the [[C2. Map JD Requirements to Supporting Evidence#B. Reference Column – Standardized Format|Reference Column – Standardized Format]] defined below.
4. Record the **Original Text** exactly as it appears in the source table.
5. Assign the correct **CV Position** using the mapping rules defined in [[C2. Map JD Requirements to Supporting Evidence#C. CV Position Column – Standardized Values|CV Position Column – Standardized Values]] below.
6. If one requirement is supported by multiple pieces of evidence, create one row per reference (1-to-many relationship).
7. **Conduct a deduplication review**: Reassign evidence that appears in multiple requirements according to the [[C2. Map JD Requirements to Supporting Evidence#E. Deduplication and Best-Fit Decision Rules|Deduplication and Best-Fit Decision Rules]].
8. Produce the SharePoint-ready export (D.2 format).

---

## B. Reference Column – Standardized Format

The Reference column must follow the structure of the Profile_Reference_Workbook.xlsx tables.

| Source Table         | Reference Format Example                       | When to Use                                      |
| -------------------- | ---------------------------------------------- | ------------------------------------------------ |
| tbl_STARs            | Tbl_STARs > STAR_ID 5                          | When referencing the overall STAR story          |
| tbl_STAR_Actions     | Tbl_STAR_Actions > Action_ID 5-3               | Most common – specific action taken              |
| tbl_STAR_Results     | Tbl_STAR_Results > Result_ID 5-R2              | When referencing a quantified outcome            |
| tbl_STAR_Competences | Tbl_STAR_Competences > CompetenceEntry_ID 5-C1 | When referencing demonstrated behaviour          |
| tbl_STAR_Attributes  | Tbl_STAR_Attributes > AttributeEntry_ID 5-A2   | When referencing personal attributes             |
| tbl_Responsibilities | Tbl_Responsibilities > Responsibility_ID A-R3  | When referencing a generic role responsibility   |
| tbl_Education        | Tbl_Education > Education_ID EDU-3             | When referencing a degree or executive education |
| tbl_Languages        | Tbl_Languages > Language_ID LANG-3             | When referencing language proficiency            |

**Rules:**

- Always use the exact table name and ID from Profile_Reference_Workbook.xlsx.
- Prefer the most specific level possible (e.g., Action_ID over STAR_ID when a specific action supports the requirement).
- Never invent IDs. Only use values that exist in the workbook.

## D. Evidence Strength Assessment Criteria

When evaluating how well a piece of evidence supports a specific requirement, use the following criteria:

| Strength       | Definition                                                                 | Guidance |
|----------------|----------------------------------------------------------------------------|----------|
| **Very Strong**    | The evidence directly and specifically demonstrates the core behaviour or outcome described in the requirement. | Prioritise as primary evidence |
| **Strong**         | The evidence clearly shows the required behaviour or skill, even if the context is not perfectly aligned. | Good supporting evidence |
| **Moderate**       | The evidence is related to the requirement but is indirect, partial, or requires interpretation. | Use only when stronger evidence is unavailable |
| **Weak**           | The evidence is tangential or only loosely connected to the requirement. | Generally avoid |

**Rule:** Favour **Very Strong** and **Strong** evidence. Only use Moderate or Weak evidence when no stronger options exist for that requirement.

---

## E. Deduplication and Best-Fit Decision Rules

Each piece of evidence should normally be assigned to **only one requirement** — the one where the connection is strongest and most specific.

**Primary Rule:**
> Assign evidence to the requirement where it demonstrates the **highest relevance and specificity**. Avoid duplicating the same evidence across multiple requirements unless there is a clear and justified reason.

**Decision Hierarchy (apply in this order):**

1. **Specificity** — Choose the requirement that the evidence addresses most directly and specifically.
2. **Priority** — If the evidence fits two requirements with similar strength, assign it to the requirement with the higher priority (**Core > Important > Nice-to-Have**).
3. **Sequence** — If still unclear, assign the evidence to the requirement that appears earliest in the list.
4. **Exception** — Only duplicate evidence across requirements in exceptional cases. Document the reason briefly in the Notes.

**Process Check:**
After the initial mapping, perform a deduplication review and reassign evidence according to the rules above.

---

## F. Handling Requirements with Weak or No Direct Evidence

When a requirement has limited or no strong supporting evidence:

- Be transparent. Do **not** force weak or indirect evidence into the mapping simply to fill a gap.
- Clearly indicate in the output that the requirement has **limited or no direct evidence**.
- Consider whether the gap can be addressed later in the **Profile** section or the **Motivational Letter**.
- Maintain the overall quality of the mapping by avoiding weak evidence.

---

## G. Output

**Format**  
SharePoint-Ready Export

**Columns (in exact order):**

Lead · Lead:ID · Requirement Order · Rank · Requirement Group · Requirement · Requirement Description · Skills · Reference · Original Text · CV Position

**Rules:**

- One row per reference (multiple rows allowed per requirement).
- Sort by Requirement Order ascending.
- Reference and Original Text must be copied exactly from Profile_Reference_Workbook.xlsx.
- Leave ID blank (auto-generated by SharePoint).

---

### Notes & Quality Rules

- This step must remain **truthful**. Only map a requirement if there is genuine, defensible evidence in the workbook.
- Prioritise evidence that is **Very Strong** or **Strong** according to the Evidence Strength Assessment Criteria.
- Apply the Deduplication and Best-Fit Decision Rules to avoid unnecessary duplication across requirements.
- When evidence is indirect or partial, still record it but note the limitation where relevant.
- C2 is the single source of truth for all subsequent tailoring steps (C3 – CV Bullets, C4 – Actual Skills, C5 – Profile).

## A. CV Source Selection

### A.1 Folder and Naming Convention

**Folder:**
`OneDrive > Documents > Obsidian Vault > JobSearch Camunda > Group CVs`

**File naming convention:**
`Reginaldo_Silva_Jr_CV - [Primary]-[Secondary].docx`

Where Primary and Secondary are group codes from the priority table below.
Pure-group CVs (single code, no secondary) may also exist:
`Reginaldo_Silva_Jr_CV - [Primary].docx`

### A.2 Group Code and Priority Table

Based on the JD Group Decision Rules table:
![[B4. Translate Requirements to Areas of Expertise and Define JD Groups#JD Group Decision Rules]]


Lower priority number = higher priority in tie-breaking.

### A.3 Selection Rules

Apply in order — stop at the first match found.

**For a Pure-Group role (Primary only, no Secondary):**

| Step | Look for | If multiple, select |
|---|---|---|
| 1 | `[P].docx` (pure group file) | — only one can exist |
| 2 | `[P]-*.docx` (P as primary, any secondary) | Lowest priority number for `*` |
| 3 | `*-[P].docx` (P as secondary, any primary) | Lowest priority number for `*` |
| 4 | No match | Flag: `No CV available for [P] — using closest: [file used]` |

**For a Dual-Group role (Primary + Secondary):**

| Step | Look for | If multiple, select |
|---|---|---|
| 1 | `[P]-[S].docx` (exact match) | — only one can exist |
| 2 | `[S]-[P].docx` (reversed — covers both groups, note emphasis difference) | — only one can exist |
| 3 | `[P]-*.docx` where `*` ≠ S | Lowest priority number for `*` |
| 4 | `*-[S].docx` or `[S]-*.docx` | Lowest priority number for anchor |
| 5 | No match | Flag: `No CV available for [P]-[S] — using closest: [file used]` |

**Always record the CV file actually used in the D.1 block header.**
When a fallback is used, note it explicitly and flag as `⚠️ Fallback CV — initial score may be conservative`.

## B. Input Documents

For each Job Lead, read the following before scoring:

1. **Group CV** — selected per Section A. Read the full DOCX file via MCP.
2. **Job Requirements** — read from `Job Requirements` table in `[[Job Hunting Lists.xlsx]]`, filtered by the Job Lead: ID. This contains all requirements extracted in Step 5.
3. **JD `.md` file** (optional) — fetch from
   `OneDrive > Obsidian Vault > JobSearch Camunda > Job Descriptions`
   only when a requirement needs contextual clarification that the requirement table alone does not resolve.

Do **not** use the `Profile_Reference_Workbook.xlsx` at this stage. Profile evidence mapping belongs to the tailoring steps. The group CV is the sole source for scoring here.

## C. Scoring Rules

### C.1 Match Strength Scale

| Match Strength | Meaning | Score range |
|---|---|---|
| Excellent | CV addresses this requirement fully and explicitly, with strong keywords | 9–10 |
| Very Strong | CV addresses it clearly; minor phrasing or depth gap | 7–8 |
| Good | CV covers it partially; evidence is present but not prominent | 5–6 |
| Weak | CV has tangential or indirect evidence only | 2–4 |
| No Match | No relevant evidence in the CV | 0–1 |

Match Strength and Score must be **consistent** — do not assign "Excellent" with a score of 6 or "Weak" with a score of 8.

### C.2 Per-Requirement Score (0–10)

For each requirement row:
- Map the requirement to the strongest CV evidence (Profile section, specific bullet, Skills sidebar, Education, Languages)
- Quote or reference the exact CV text where possible - Score honestly — never inflate to make the picture look better
- If a requirement is partially covered, note the specific gap in  `Initial_Missing_Weak`
- If a requirement has no relevant CV evidence, mark `No Match` and state what is absent

**ATS keyword check:** where the JD uses specific terminology, verify whether the CV uses the same phrasing or a non-matching synonym.
Keyword mismatches reduce the score even when the underlying experience is present — note them in `Initial_Missing_Weak`.

### C.3 Overall Fit Score (0–10)

Weighted average of per-requirement scores, using Requirement Group as the weighting factor:

| Requirement Group | Weight |
| ----------------- | ------ |
| Core              | 3      |
| Important         | 2      |
| Nice-to-Have      | 1      |

`Overall Fit Score = Σ(score × weight) ÷ Σ(weight)`

Round to one decimal place. Record in `Overall Fit Score` column of the `Job Leads` table.

## D. Execution Sequence

1. Identify the Job Lead(s) to be processed.
2. For each Job Lead:
   a. Determine and read the appropriate Group CV (per Section A).
   b. Internally perform requirement extraction from the JD (as defined in [[B5. Extract Requirements from Job Description]]).
   c. Score each extracted requirement against the CV (per Section C).
   d. Calculate the Overall Fit Score (per Section C.3).
3. Repeat step 2 for all Job Leads in the batch.
4. Produce the combined D.2 export(s).

**Language note:** JDs in German are scored in their original language.
CV text and requirement descriptions are compared directly — do not translate before scoring.

### E.2 End-of-Batch SharePoint Export

Produced once after all leads in the batch are scored.
Two separate exports, each as a tab-delimited block with Copy button.

**Export 1 — Job Requirements update**
Updates existing rows in the `Job Requirements` table.
Identify rows by Lead + Requirement Order + Rank.

*(Leave blank: `ID` — auto-generated by SharePoint;
Columns (in this exact order):
`Lead  · Lead:ID  · ID · Requirement Order · Rank · Group · Requirement · Requirement Description · Skills · Initial Match Strength · Initial Key Strengths · Initial Missing / Weak · Initial Score`

**Export 2 — Job Leads Overall Fit Score update**
Updates the `Overall Fit Score` column in the `Job Leads` table.

Columns: `Title · Overall Fit Score`

**Rules (both exports):**
- Tab-delimited with RFC 4180-style quoting for multi-line fields
- One header row, one data row per requirement (Export 1) or per lead (Export 2)
- Leads sorted by SharePoint ID descending (highest ID first)
- Requirements sorted by Requirement Order ascending within each lead
- Two buttons: "Copy all" (header + data) and "Copy data only"

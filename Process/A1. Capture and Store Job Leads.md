## Purpose

Capture a Job Lead into RoleProof (`job_leads`, `status: captured`) with as much structured, usable data populated automatically at capture time as can be reliably obtained — without waiting for the B-phase screening pipeline and without burning B/C-phase LLM calls on leads that may never be promoted.

This replaces the retired SharePoint / Power Automate flow (see the old `A1. Store Job Leads` note, Process/, pending retirement once this version is promoted). **Decision (2026-07-23): the bookmarklet transport is retired.** LinkedIn's own CSP blocks any `fetch()`-based capture from within its page (section A.2), and most leads originate on LinkedIn — patching around that a second time (form-POST, browser extension) was considered and set aside in favour of AI-driven capture, which sidesteps the problem structurally rather than working around it, and is the only option that also covers capturing from mobile. Two capture paths remain:

1. **Manual paste** — `/roleproof/capture`, unchanged, same-origin form submit via `createLeadAction`. Never had a transport problem; stays as the always-available fallback.
2. **AI-driven capture** — an agent given a job posting URL reads it and creates the lead on the user's behalf. This is now the primary path. **Transport decision (2026-07-23):** the browser side of this runs on Playwright (via `playwright-mcp`), not an interactive session's own browser extension — see A.3. That choice is specifically about running unattended later; it doesn't change anything in sections B/C/D, which only care that `{title, url, markdown, candidateLinks, company, city, remote, formatSignals}` arrives at `createLead()` somehow.

## A. Capture Sequence — AI-driven path

1. **Trigger.** User shares a job posting URL with an agent session — from a computer or a phone, since this is just handing over a link, not running a browser bookmarklet.
2. **Read, not scrape-and-fetch.** The agent navigates to the URL using a real browser it controls (section A.3), authenticated as the user, and reads the rendered page — title, full text, and every anchor `href` on the page. This is a DOM/accessibility-tree read, not a script executing inside the page's own document — see A.1 for why that distinction is what makes this immune to LinkedIn's CSP.
3. **Candidate off-site links.** From the collected hrefs, filter out `linkedin.com`/`licdn.com`/`lnkd.in` domains and unwrap any `linkedin.com/safety/go/?url=<target>` redirect wrapper first (same logic as section B.3, now applied by the agent reading the page instead of injected bookmarklet JS).
4. **Extraction, inline.** Since the agent already has the full JD text in context from step 2, it extracts `company`, `city`, `remote`, and `formatSignals` itself as part of the same read — no separate model call needed for this path (see revised section C).
5. **Submit.** The agent POSTs `{ title, url, markdown, candidateLinks, company, city, remote, formatSignals, token }` as plain JSON to `/api/ingest` — same endpoint and shape originally designed, no form-POST workaround required. The request must not originate from a script running inside the posting page's own document (that's the wall being avoided): in production this is simply the agent's own HTTP call to the deployed public URL; for local pre-deployment testing, it's a call from the Playwright-controlled browser's own process, or from a tab on RoleProof's own origin — either way, it's never LinkedIn's page making the call.
6. **Deterministic enrichment (Section B)** — runs synchronously in `createLead()` exactly as before, no model call: URL cleanup, `jobPostLink` derivation, ATS detection. Unaffected by which path submitted the data.
7. **Persist.** Row is `status: captured`, ready for B1 whenever the user chooses to screen it.

### A.1 Why this is immune to LinkedIn's CSP

`Content-Security-Policy`'s `connect-src` directive restricts network calls (`fetch`/`XHR`/WebSocket) made by scripts executing *inside a given document*. It has no jurisdiction over a separate process reading that document's rendered content from outside (browser automation via the DevTools protocol, not an in-page script) or over a second, unrelated HTTP call made by that separate process to a different server entirely. The agent reading the LinkedIn page and the agent submitting to `/api/ingest` are, from LinkedIn's server's point of view, not connected to each other at all — there's no page script making a cross-origin call for CSP to catch.

Validated live (2026-07-23) using Claude in Chrome driving an interactive session's own browser: read a real gated LinkedIn posting, then POSTed the full JSON payload from a tab on `https://localhost:3000` — 200 OK, no CSP error. That confirmed the concept. It is not, however, the transport going forward — see A.3.

### A.2 Bookmarklet — retired

Confirmed live via DevTools console: *"Fetch API cannot load .../api/ingest. Refused to connect because it violates the document's Content Security Policy."* This is enforced by LinkedIn's own response headers — nothing about our server's cert, scheme, or CORS configuration could ever fix it, which is also why the original SharePoint-era bookmarklet used `window.open()` (a navigation, exempt from `connect-src`) rather than `fetch()` in the first place; the same wall existed then and was worked around the same way each time. Rather than repeat that workaround a third time (this time as a hidden form POST), the bookmarklet is retired. `app/roleproof/capture/page.tsx`'s bookmarklet generator can be removed as implementation cleanup — not required for AI-driven capture to work, just dead code once it's gone.

### A.3 Browser mechanism: Playwright (`playwright-mcp`), not the interactive extension

Claude in Chrome proved the concept but is fundamentally an interactive-session feature — it drives the user's live daily browser through an extension, with no unattended/server equivalent. The stated goal is capture that eventually runs headless on a server; [`playwright-mcp`](https://github.com/microsoft/playwright-mcp) is built for exactly that (a real Playwright-controlled browser, headed or headless, exposed as MCP tools any client — including a Claude Code session — can drive), so it's the mechanism going forward, not just for later but for local work from now on.

**Browser: Microsoft Edge**, via `--browser msedge` (or `PLAYWRIGHT_MCP_BROWSER=msedge`) — Playwright MCP supports channel-based browsers directly, matching Reggie's daily driver rather than defaulting to a bare Chromium.

**Login/session (deliberately interim — this is the pragmatic step, not the finished design):**

- For now: launch the browser **non-headless** locally. Reggie logs into LinkedIn manually, once, in that visible window.
- Playwright MCP already keeps a persistent per-project browser profile by default (`ms-playwright/mcp-{channel}-{workspace-hash}` in the platform cache dir, or a custom path via `--user-data-dir`) — check whether that alone makes the session stick across runs before building anything extra. Only fall back to a manual `storageState` (cookies + localStorage) export/import, saved under the existing gitignored `.storage/` convention, if the default profile isn't enough. Either way, whatever file ends up holding the session is live authentication material, equivalent to a saved login — treat it that way.
- Subsequent captures reuse that saved `storageState` when creating a new browser context — no repeated manual login, no credentials typed or stored by the agent itself.
- **Explicitly deferred, not solved:** a proper credential store (secrets manager, refresh handling, headless server deployment of this same session). This section exists so that deferral is visible and doesn't quietly become "permanent" — revisit before this moves to a server.

## B. Deterministic Enrichment (code, no model call)

### B.1 URL cleanup → `sourceUrl` vs `jobPostLink`

- `sourceUrl` = the raw captured URL, untouched. This is the audit trail and, when the user captures from the LinkedIn posting itself, the reference used later to refresh `posted_days` / `applicant_count` without re-running B1–B6.
- `jobPostLink` = `sourceUrl` with known tracker query parameters stripped. This is the link used to apply and the one B4 (or B.2 below) reads to identify the ATS.

**Strip by parameter key, not by literal string match** — the CI note's own examples show the values vary (`igbTracker=926613858` — the number changes every capture), so matching has to be key-based:

Deny-list (case-insensitive, strip if present, regardless of value):
`utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content`, `source`, `sourceType`, `sourcetype`, `Codes`, `feedId`, `igbTracker`, `trk`, `trackingId`, `li_fat_id`, `refId`, `midToken`, `liCampaign`

Rule: parse the URL, drop any query parameter whose key matches the deny-list above (case-insensitive), keep every other parameter untouched (job/requisition IDs like `?jobId=` or `?req=` must survive — they're needed to actually load the posting). Re-serialize the URL. If stripping leaves an empty query string, drop the trailing `?`.

### B.2 ATS detection from `jobPostLink` domain

Match the `jobPostLink` hostname against known ATS domain patterns. If matched, populate `atsSystem` immediately at capture. If no match, leave `atsSystem` null — B4 still runs its own (LLM-based, JD-content) detection as the fallback for anything not caught here, so this is additive, not a replacement.

| Hostname pattern | ATS |
|---|---|
| `myworkdayjobs.com`, `myworkday.com` | Workday |
| `greenhouse.io`, `boards.greenhouse.io` | Greenhouse |
| `lever.co`, `jobs.lever.co` | Lever |
| `smartrecruiters.com` | SmartRecruiters |
| `icims.com` | iCIMS |
| `successfactors.com`, `career.sap.com` | SAP SuccessFactors |
| `taleo.net` | Taleo |
| `personio.de`, `personio.com` | Personio |
| `bamboohr.com` | BambooHR |
| `avature.net` | Avature |
| `cornerstoneondemand.com`, `csod.com` | Cornerstone |
| `oraclecloud.com` | Oracle Fusion/Cloud HCM |
| `onlyfy.jobs` | Onlyfy (formerly softgarden) |

Table is extendable in place — add a row rather than opening a CI item every time a new ATS domain shows up.

### B.3 Off-site apply link candidates (LinkedIn capture)

Most leads originate on LinkedIn, and this is the case B.1/B.2 don't cover as written: on a LinkedIn job view page, the captured `url` is always `linkedin.com` — it never resolves to a known ATS domain, so `jobPostLink`/`atsSystem` would silently fall back to the LinkedIn link itself. The actual apply destination (the direct company/ATS link, often already carrying `utm_source=linkedin`-style trackers per the CI note) only exists as an off-site link elsewhere on the page. On a logged-in view, LinkedIn routes it through a `linkedin.com/safety/go/?url=<encoded target>` redirect wrapper rather than linking off-site directly — unwrap that `url` param first.

Don't target a specific "Apply button" selector for this — LinkedIn's markup for it varies (Easy Apply vs off-site apply, logged-in vs guest view) and changes without notice, so a hardcoded selector is fragile and fails silently. Instead, collect every anchor `href` on the page, filter out any whose hostname is `linkedin.com`, `licdn.com`, `lnkd.in`, or another LinkedIn-owned domain (unwrapping the redirect wrapper above first), and pass the result as `candidateLinks: string[]` to `createLead()`. Under the AI-driven path (section A) this collection happens as part of the agent's own page read; the server-side handling is unchanged either way:

- **Server (`createLead()`):** run each candidate through the same hostname match as B.2. The first candidate matching a known ATS domain wins — it becomes `jobPostLink` (after B.1 tracker-stripping) and sets `atsSystem`. No match → fall back to today's behavior (`jobPostLink` = cleaned `sourceUrl`).
- `sourceUrl` is unaffected either way — always the page actually captured from (the LinkedIn URL, in this case), still the reference for the freshness/applicant-count refresh.
- Easy Apply postings (no off-site link exists at all) degrade gracefully to the fallback — expected, not a bug.

## C. Extraction: agent-supplied first, DeepSeek fallback second

**This is extraction only — no judgment, no scoring, no recommendation.** It fills fields; it does not decide anything, regardless of which path produces it. Explicitly out of scope here: roadblocks, misalignments, skill ratings, fit scoring (B2–B6), and the format/headshot/DEI recommendation (C1) — those stay exactly where they are, gated behind B6 promotion, so cost only gets spent on leads that survive screening.

**Precedence in `createLead()`:** if `company`, `city`, `remote`, or `formatSignals` arrive already populated on the input (from the AI-driven path, which read the full JD and filled these directly, or from a human typing into the manual form), use them as given — skip the model call for whichever fields are already present. Only fall back to a small forced-schema DeepSeek call, over the captured `markdown`, for whatever's still missing — the case that matters here is a plain manual paste with no agent involved, where nothing but the JD text itself was provided.

**Extraction rules (apply whether it's the agent doing this inline in step A.4, or the DeepSeek fallback call):**

Read the job description text and extract only what is explicitly present or unambiguously inferable. Do not guess. Leave a field null/unspecified rather than invent a value.

- `company` — the hiring company's name.
- `city` — the primary work location's city.
- `remote` — one of `on-site`, `hybrid`, `remote`, `unspecified`. Only set `remote` or `hybrid` if the posting says so explicitly; default to `unspecified` rather than assume `on-site`.
- `formatSignals` — verbatim quotes (not paraphrased) of any explicit application-format instructions found in the text: CV length/page limits, required file type, file naming convention, cover-letter requirement, mention of a photo/headshot, language of application, HR/Talent Acquisition contact name. Concatenate whatever is found as short quoted fragments; leave empty if nothing explicit is stated. **This is raw material for C1, not a decision** — C1 still does its own market-practice/DEI reasoning against this text when the lead is promoted; this pass only saves it from having to re-read the full JD from scratch.

The DeepSeek fallback call uses the same forced tool-call pattern as every other step (`runStructured`), one bounded retry on Zod validation failure, logged to `llm_calls` like every other step.

## D. Schema Changes Required

None of the following exist yet — confirmed against `lib/db/schema.ts`, not assumed from the architecture docs (those describe a company/office upsert at capture that isn't actually implemented).

- `job_leads.remote` — new text/enum column (`on-site` / `hybrid` / `remote` / `unspecified`).
- `job_leads.jdText` — new text column. Today the JD only exists as a markdown blob in Storage (`rawJdPath`); this mirrors it into Postgres so it's queryable. Keep `rawJdPath`/Storage write as-is for the audit copy — additive, not a replacement.
- `job_leads.formatSignals` — new text column for the C section output above.
- Populate at capture, using columns that already exist but that `createLead()` currently never sets: `companyId` (resolve/create against `companies`, keyed on the extracted or manually-typed `company` name), `jobPostLink`, `atsSystem`.
- `CaptureInput` (`lib/pipeline/capture.ts`) needs `remote?`, `formatSignals?`, and `candidateLinks?: string[]` added alongside the existing `company?`/`city?` — the AI-driven path supplies all of these directly; today's type only has room for `company`/`city`.

`sourceUrl` and `jobPostLink` already exist as separate columns in the schema — no migration needed for those two, just logic to populate `jobPostLink`. No API/route contract change needed: `/api/ingest` already accepts a JSON body, which is exactly what the AI-driven path sends — the form-POST variant considered in the retired A.2 approach is no longer necessary.

## E. Output

A `job_leads` row at `status: captured` with `title`, `company`/`companyId`, `city`, `remote`, `sourceUrl`, `jobPostLink`, `atsSystem` (when detectable), `jdText`, `rawJdPath`, and `formatSignals` all populated without any manual re-entry — ready for the user to prioritize and, whenever they choose, send into B1.

## Notes / Continuous Improvement

- **Refresh without re-screening.** The user may sit on a captured lead for several days before starting tailoring; by then `posted_days` / `applicant_count` are stale. A follow-up action should let the user re-run capture against the same lead's `sourceUrl` to refresh just those two fields (B1's freshness/saturation inputs) without re-running B2–B6. Not spec'd here — worth its own CI item once this one ships.
- **Retiring the old note.** The old `A1. Store Job Leads` note (`Process/`, Power Automate/SharePoint flow) has a distinct title now, so no collision — but it's still the `Valid` note today. When this draft is promoted, move it to `Past Versions` with `pr-retired` set (same pattern as `B6`/`C4`), and update `docs/PIPELINE.md`'s A1 row (currently points at the old filename) to reference this note.
- **Local testing, pre-deployment.** This Cowork sandbox can't reach `localhost:3000` on the user's machine directly (separate networks), so validating the AI-driven path locally means the browser tool either drives the existing manual-paste form on `https://localhost:3000/roleproof/capture`, or opens a tab on that same origin and submits from there — never a raw request from this sandbox. Once RoleProof is deployed (Vercel, public URL), an agent's own HTTP call to the production `/api/ingest` is the clean version of this and needs no browser-driven workaround at all.
- **Mobile capture.** This was the original ask this CI item couldn't otherwise satisfy — a browser extension (the alternative considered and set aside) has no mobile equivalent; sharing a URL into an agent session works identically from a phone.
